## Загальна класифікація периферійних інтерфейсів

Цей розділ — **орієнтир перед усіма лекціями**. Конспект 2025 розглядає конкретні стандарти по черзі; тут зведено загальні ознаки та відповідність лекціям курсу. Терміни **UK / EN** — як у datasheet (*simplex*, *half-duplex*, *full-duplex*, *serial*, *parallel*, *asynchronous*, *synchronous*).

### За напрямком передачі (duplex)

| Режим (UK / EN) | Зміст | Приклади в курсі |
|-----------------|--------|------------------|
| **Симплекс** / *simplex* | Дані лише в один бік | Датчик → MCU (один канал), де зворотний зв’язок окремий |
| **Напівдуплекс** / *half-duplex* | Обидва боки, але не одночасно | RS-485, CAN, I²C (чергування на шині) |
| **Повний дуплекс** / *full-duplex* | Одночасна передача в обидва боки | RS-232 (окремі лінії TX/RX), SPI (MOSI/MISO), USB (різні endpoints) |

### За способом передачі бітів (serial vs parallel)

| Тип (UK / EN) | Ознаки | Приклади | Лекції |
|---------------|--------|----------|--------|
| **Послідовний** / *serial* | Біти один за одним по одній або кількох лініях | RS-232C, USB, CAN, SPI, I²C, MIL-1553B | 1–6, 4 |
| **Паралельний** / *parallel* | Кілька бітів одночасно + службові лінії | Centronics, IEEE-488 (GPIB) | 9 |
| **Магістраль** / *bus* | Спільна шина, адресація / арбітраж | ISA, PCI, PCIe, CAN, I²C | 3, 7–8 |

### За синхронізацією

| Тип (UK / EN) | Ознаки | Приклади | Лекції |
|---------------|--------|----------|--------|
| **Асинхронний** / *asynchronous* | Стартовий/стоповий біт, узгоджена швидкість (baud) | RS-232C, UART через USB-UART | 1 |
| **Синхронний** / *synchronous* | Загальний тактовий сигнал або вбудована синхронізація протоколу | SPI, I²C, CAN, USB (SOF), PCI/PCIe | 2–6, 8 |

### За топологією з’єднання

| Топологія (UK / EN) | Ознаки | Приклади | Лекції |
|---------------------|--------|----------|--------|
| **Point-to-point** / *вузол–вузол* | Два вузли | RS-232 (класично), USB (host–device) | 1–2 |
| **Шина** / *bus* | Спільна лінія, адресація або арбітраж | CAN, I²C, RS-485 | 1, 3, 6 |
| **Зірка** / *star* | Центральний вузол | USB (hub), PCIe (root complex) | 2, 8 |
| **Master–slave** / *ведучий–ведений* | Один ініціює обмін | SPI, I²C | 5–6 |

**Приклади EN-формулювань:** I²C — *synchronous, multi-master/multi-slave, single-ended, serial communication bus*; SPI — *synchronous, full-duplex, serial*; UART — *asynchronous, serial, point-to-point*.

### За роллю в системі

| Роль | Приклади | Лекції |
|------|----------|--------|
| **Комунікаційний порт** | RS-232, USB CDC-ACM | 1–2 |
| **Польова / промислова шина** | CAN, RS-485, MIL-1553B | 1, 3–4 |
| **Внутрішня магістраль ПК** | ISA, PCI, PCIe | 7–8 |
| **Інтерфейс накопичувача** | ATA/SATA, SCSI, NVMe | 10, 12 |
| **Введення / виведення** | HID, принтер, дисплей (DP/HDMI) | 13–14 |
| **Аудіо / мовлення** | USB Audio, звукова карта | 15 |
| **Мережевий доступ до периферії** | Ethernet (NIC), принтер по IPP, NAS, LXI | доповнення; лек. 8–9, 14 |

### Мережеві периферійні інтерфейси (огляд)

**Ethernet не входить до офіційних 15 лекцій** конспекту 2025 — детальний розбір TCP/IP, маршрутизації та Wi‑Fi належить курсу **«Комп’ютерні мережі»**. Для PPID достатньо зрозуміти, **де мережа стоїть поруч із «класичними» периферійними інтерфейсами** і на якому рівні працює програміст.

#### Ethernet і мережева карта (NIC)

| Рівень | Що це | Аналогія в курсі PPID |
|--------|--------|------------------------|
| Фізичний | Кручена пара (100M–10G+), оптика; **1 / 2.5 / 10 / 25 GbE** | Як лінія RS-232 або USB, але для **пакетної мережі** |
| Пристрій у ПК | **NIC** на **PCIe** (лекція 8) | Периферія на магістралі, як відеокарта чи NVMe |
| Драйвер ядра | Драйвер мережевої карти (`e1000`, `igb`, `mlx5` …) | Як USB Mass Storage driver |
| API для програми | **Сокети** (BSD sockets), не прямий доступ до кадрів | Як запис файлу на `E:\` замість USB-транзакцій |

Програма з `socket()` **не є** драйвером Ethernet — так само, як запис на USB-флешку не є USB device driver (лаб. 3).

#### Периферія «через мережу»

| Пристрій | Протокол / підхід | Зв’язок з лекціями |
|----------|-------------------|---------------------|
| Мережевий принтер | **IPP**, JetDirect, SMB | лекція 14 (виведення) |
| NAS, мережеве сховище | NFS, SMB, **iSCSI** | лекції 10, 12 (накопичувачі) |
| Лабораторний прилад | **LXI** (Ethernet + HTTP/VISA) | лекція 9 (заміна GPIB) |
| IP-камера, VoIP | RTP/RTSP поверх IP | лекції 13–15 (оглядово) |

#### Промислові мережі на базі Ethernet (лише згадка)

Поруч із **CAN** (лекція 3) у factory automation використовують:

- **EtherCAT**, **PROFINET**, **Ethernet/IP** — реальний час і циклічний обмін поверх Ethernet;
- це **не** звичайний офісний TCP/IP, а спеціалізовані протоколи поверх тієї ж фізичної мережі.

#### Бездротові інтерфейси (лише місце в класифікації)

| Інтерфейс | Примітка для PPID |
|-----------|-------------------|
| **Wi‑Fi** | Бездротовий NIC; стек у ОС (mac80211 у Linux); окремий курс |
| **Bluetooth** | Гарнітури, HID (лекція 15); не низькорівневий UART |

#### Серверні інтерфейси (огляд, поза лабами)

**Fibre Channel**, **InfiniBand** — високошвидкісні мережі для СХД та HPC; конкурують із **Ethernet + NVMe-oF** у дата-центрах. Достатньо знати, що це **мережевий доступ до накопичувачів/пам’яті**, а не локальна шина типу SATA.

**Чому немає лабораторної на Ethernet у PPID:** навчальна мета практикуму — послідовний порт і модель USB; мережеве програмування (сокети, протоколи) — інша дисципліна та інший тип звіту.

**Питання для самоперевірки (мережі, огляд):**

1. Чим NIC на PCIe схожий на USB-контролер з точки зору архітектури ПК?
2. Чому програма, що друкує через IPP, не замінює драйвер мережевої карти?
3. У чому відмінність між CAN-шиною в автомобілі та EtherCAT у промисловості (одне речення)?

## Загальні доповнення (актуальні для всього курсу)

### Рівні програмної підтримки периферії сьогодні

```text
Периферійний пристрій
    → firmware (вбудоване ПЗ пристрою)
    → драйвер ядра ОС (kernel driver)
    → API операційної системи
    → прикладні бібліотеки (pyserial, libusb, HID API)
    → програма користувача
```

Студент бакалавра на лабораторних зазвичай працює на **рівні API ОС** або **user-space бібліотек**, а не пише kernel driver. Це нормально й відповідає меті курсу — зрозуміти логіку взаємодії, а не замінити розробників драйверів Linux/Windows.

### Теми, яких майже немає в конспекті 2025

| Тема | Чому важливо |
|------|----------------|
| USB-UART (FTDI, CH340, CP2102) | Сучасна реалізація «COM-порту» на ПК |
| USB-C: PD, CC-піни, DRP, alt modes, типи кабелів | Один роз’єм — USB, DisplayPort, Thunderbolt |
| USB 3.2 / USB4 / USB4 2.0 / Thunderbolt 4 | Еволюція після USB 3.0 з лекції 2 |
| PCIe Gen4/5/6, NVMe | Заміна частини SATA/SCSI в ПК і серверах |
| eMMC / UFS | Накопичувачі в мобільних і embedded |
| DisplayPort 2.1 / HDMI 2.1 | Сучасне виведення відео на монітори (лекція 14) |
| Ethernet, NIC, LXI, промисловий Ethernet | Мережевий доступ до периферії (огляд у класифікації) |
| CXL | Пам’ять і accelerators поверх PCIe (огляд) |
| Платформи embedded (RPi, ESP32, Jetson) | Зв’язок лекцій з реальними платами (таблиця нижче) |

### Платформи embedded: які інтерфейси де

Для спеціалізації **КСМ** лекції PPID найкраще «приземляти» на платформи, з якими ви зіткнетеся в проєктах IoT, робототехніки та embedded Linux.

| Платформа | Клас | Типові інтерфейси з курсу PPID | Доступ у ПЗ (приклад) |
|-----------|------|--------------------------------|------------------------|
| **Raspberry Pi 4/5** | Linux SBC | **USB** host, **Ethernet** (GbE), **HDMI**, **UART/SPI/I²C** на 40-pin GPIO, **PCIe** (Pi 5, x1), SD/eMMC | `/dev/ttyAMA0`, `/dev/spidev*`, `/dev/i2c-*`, `lsusb`, `ip link` |
| **ESP32** (Espressif) | MCU + Wi‑Fi/BT | **UART**, **SPI**, **I²C**, **CAN (TWAI)**, **ADC**, USB-CDC (S2/S3/C3) | Arduino/ESP-IDF: `Serial`, `SPI`, `Wire`; Wi‑Fi — окремий стек |
| **Jetson Nano / Orin Nano** | Linux embedded (ARM) | **USB 3**, **Gigabit Ethernet**, **HDMI/DP**, **MIPI CSI** (камера), GPIO **UART/SPI/I²C**, **M.2** (Wi‑Fi, часто PCIe) | JetPack, `device tree`, GStreamer для камери |
| **STM32** (Nucleo, Blue Pill) | MCU | **USART**, **SPI**, **I²C**, **CAN**, **USB device**, **ADC/PWM** | HAL/LL: `HAL_UART`, `HAL_SPI`; CubeMX |
| **Arduino Uno / Mega** | MCU (навчальна) | **UART** (через USB-UART на платі), **SPI**, **I²C** | `Serial`, `SPI`, `Wire` |
| **BeagleBone Black** | Linux SBC | **USB**, **Ethernet**, GPIO **UART/SPI/I²C**, **eMMC** | `/dev/ttyO*`, Device Tree overlays |

**Як читати таблицю разом із лекціями:**

| Лекція | Де побачити на практиці |
|--------|-------------------------|
| 1 RS-232/UART | Консоль Pi (`/dev/serial0`), прошивка STM32/ESP32, Arduino `Serial` |
| 2 USB | Флешка в Pi, веб-камера; ESP32-S3 — USB CDC |
| 3 CAN | MCP2515 на SPI + Pi; вбудований TWAI в ESP32; OBD-II адаптер |
| 5 SPI | Датчик/flash на ESP32; `spidev` на Pi |
| 6 I²C | RTC, OLED, IMU на будь-якій платі; `i2cdetect` на Pi |
| 8 PCIe | Pi 5 (NVMe HAT), Jetson M.2; відеокарта в ПК |
| 10–12 | SD-карта Pi, NVMe на Pi 5 / ПК |
| 11 | ADC ESP32/STM32, I²S аудіо на Pi |
| 13–14 | USB HID миша; HDMI монітор до Pi/Jetson |
| Мережі (доповнення) | `ssh` на Pi/Jetson по Ethernet; MQTT з ESP32 по Wi‑Fi |

**Питання для самоперевірки (embedded):**

1. Чим відрізняється UART на GPIO Raspberry Pi від USB-CDC на ESP32-S3 з точки зору драйверів у Linux?
2. Навіщо CAN-контролер MCP2515 часто підключають по SPI, а не «напряму» до GPIO?
3. Які інтерфейси лекцій 1–2 ви використовуєте в лабах PPID, а які — на типовій платі ESP32 без ПК?

## Лекція 1. Інтерфейс RS-232C та драйвери комунікаційних портів

> **Перед цією лекцією:** розділ «Загальна класифікація периферійних інтерфейсів» — RS-232C як перший приклад **послідовного асинхронного point-to-point** інтерфейсу.

**Базовий матеріал:** класифікація RS-232C, формат слова, UART, flow control, адреси COM1–COM4, мікросхеми 8250/16550 (конспект 2025, лекція 1).

**Що додати / оновити:**

- На сучасних ПК **фізичний DB9 на материнській платі** — рідкість; типовий шлях: **USB-UART адаптер** → віртуальний COM (`COM3`, `/dev/ttyUSB0`).
- Для лабораторних без залізного порту: **віртуальна пара COM** (com0com, VSPD на Windows; `socat` на Linux/macOS).
- **RS-485** — поширений нащадок у промисловості (довші лінії, шина, не point-to-point як RS-232).
- Програмний доступ: **pyserial** (Python), Win32 API (`CreateFile`, `DCB`) — замість прямого звернення до портів I/O `3F8h`.
- RS-232 **залишається актуальним** як модель асинхронного обміну та в промисловій/automation галузі, не як основний порт ПК.

**Практичний контекст:** консольний порт мережевого обладнання, PLC, налагодження MCU через USB-UART; **Raspberry Pi** `/dev/ttyAMA0` або `/dev/serial0`, **ESP32/STM32** — USART у прошивці, **Arduino** — `Serial` через FTDI/CH340.

**Приклад (кадр UART):** start bit `0` → 8 data bits (LSB first) → parity (опційно) → stop bit `1`; час біта `T = 1/baud`.

**Питання для самоперевірки (додаткові):**

1. Чим відрізняється фізичний COM-порт від віртуального COM через USB-UART?
2. Навіщо в промисловості часто обирають RS-485 замість RS-232?
3. На якому рівні стеку працює програма з бібліотекою pyserial?

## Лекція 2. Інтерфейс USB: характеристики, транзакції та драйвери

**Базовий матеріал:** топологія «зірка», endpoints, типи транзакцій, NRZI, USB 1.x–3.0, OTG (конспект 2025, лекція 2).

**Що додати / оновити:**

- **USB 3.2 Gen 2×2** (до 20 Гбіт/с), **USB4** (до 40 Гбіт/с), **USB4 Version 2.0** (до 80 Гбіт/с), інтеграція з **Thunderbolt 3/4**.
- **USB-C:** reversible connector; **CC1/CC2** (Configuration Channel) — визначення орієнтації кабеля, ролі host/device, узгодження струму (**USB PD**, до 240 Вт у специфікаціях EPR); **alternate modes** (DisplayPort, Thunderbolt).
- **Ролі порту:** host, device, **DRP** (Dual Role Port — може перемикатися), **DRD** (Dual Role Device). На ноутбуках USB-C часто DRP: заряджання від адаптера або живлення периферії.
- **Типи кабелів USB-C:** лише USB 2.0 (D+/D−); USB 3.x з додатковими парами **SSTX/SSRX**; «повнофункціональні» — PD + SuperSpeed + alt modes. Не кожен USB-C кабель дає 40/80 Гбіт/с.
- **Кодування на фізичному рівні:** USB 2.0 — **NRZI** (як у лаб. 2); USB 3.x SuperSpeed — **8b/10b**; USB4 — **128b/132b** (інший рівень, ніж NRZI з навчальної моделі).
- Класи пристроїв: **HID** (миша, клавіатура), **Mass Storage**, **CDC-ACM** (віртуальний COM), **UAC** (аудіо).
- **FireWire (IEEE 1394)** — згадати лише як історичний конкурент; у 2026 не є практичною альтернативою USB.
- **Wireless USB** — історична довідка; на практиці замінений Wi‑Fi / Bluetooth.
- Інструменти: `lsusb` (Linux), Диспетчер пристроїв / PowerShell (Windows).

**Практичний контекст:** флешки, веб-камери, USB-хаби, док-станції USB-C; **Raspberry Pi / Jetson** — USB host для периферії; **ESP32-S2/S3** — USB device (CDC). У лаб. 3 моделюється Mass Storage на рівні mock-ФС, не kernel driver.

**Типи передач (огляд):** control (endpoint 0), bulk (накопичувачі), interrupt (HID), isochronous (аудіо/відео).

**Питання для самоперевірки (додаткові):**

1. Чим USB4 Version 2.0 відрізняється від USB 3.0 за швидкістю та фізичним рівнем?
2. Що таке USB Power Delivery і навіщо піни CC у роз’ємі USB-C?
3. Чому NRZI з лабораторної № 2 описує USB 2.0, а не весь сучасний USB4?
4. Чому програма, що пише файл на диск `E:\`, не є USB device driver?

## Лекція 3. Інтерфейси CL та CAN

**Базовий матеріал:** CAN: фізичний рівень, домінантні/рецесивні біти, арбітраж, bit stuffing (конспект 2025, лекція 3).

**Що додати / оновити:**

- **CAN FD** — більший payload, вища швидкість на шині.
- **CAN XL** — наступне покоління (огляд).
- **SocketCAN** у Linux — стандартний інтерфейс для CAN у software.
- Відмінність **CAN** (шина, багато вузлів) від **RS-485** (зазвичай master-slave на одній парі).
- Автомобільні **ECU**, промислова автоматика — основні сфери CAN сьогодні.
- **Промисловий Ethernet** (огляд): **EtherCAT**, **PROFINET**, **Ethernet/IP** — доповнення до CAN/RS-485 у factory automation; деталі — у розділі «Мережеві периферійні інтерфейси» на початку доповнення.

**Практичний контекст:** OBD-II адаптери, автомобільні мережі, робототехніка; **ESP32 TWAI**, MCP2515 на SPI з Raspberry Pi, SocketCAN `can0` у Linux.

**Питання для самоперевірки (додаткові):**

1. Для чого в CAN використовується bit stuffing?
2. Чим CAN FD розширює класичний CAN 2.0?
3. Наведіть приклад застосування CAN поза автомобільною галуззю.

## Лекція 4. Інтерфейс MIL-STD-1553B

**Базовий матеріал:** військово-авіаційна шина, керований режим обміну (конспект 2025, лекція 4).

**Що додати / оновити:**

- Залишити як **нішевий** стандарт (авіація, оборона, космос).
- Коротко: **ARINC 429** (авіоніка), **SpaceWire** (космічні системи) — суміжні галузеві інтерфейси.
- Для більшості комп’ютерних інженерів — знати **призначення та місце в класифікації**, без глибокої реалізації.

**Практичний контекст:** спеціалізована авіоніка; у масовій IT-індустрії не застосовується.

**Питання для самоперевірки (додаткові):**

1. Чому MIL-1553B має сенс вивчати на оглядовому рівні в загальному курсі PPID?
2. Чим відрізняється керована шина 1553B від мультимайстерної CAN?

## Лекція 5. Інтерфейс SPI

**Базовий матеріал:** master-slave, MOSI/MISO/SCK/CS, режими CPOL/CPHA (конспект 2025, лекція 5).

**Що додати / оновити:**

- **QSPI** — швидкий доступ до зовнішньої flash (boot, firmware).
- **DMA** при обміні SPI на MCU — зниження навантаження на CPU.
- Типові платформи: **STM32**, **ESP32**, **Raspberry Pi** (SPI через `/dev/spidev`).
- **Logic analyzer** — практичний інструмент налагодження SPI.

**Практичний контекст:** датчики, EEPROM, дисплеї, радіомодулі на embedded-платах; **ESP32** — кілька SPI host; **Raspberry Pi** — `/dev/spidev0.0`, режими Mode 0–3.

**Приклад підключення:** MOSI, MISO, SCK + окремий **CS** на кожен slave-чіп.

**Питання для самоперевірки (додаткові):**

1. Навіщо лінія CS (Chip Select) при підключенні кількох slave-пристроїв?
2. Чим SPI відрізняється від I²C за топологією та швидкістю?

## Лекція 6. Інтерфейс I²C

**Базовий матеріал:** двопровідна шина SDA/SCL, адресація, open-drain (конспект 2025, лекція 6).

**Що додати / оновити:**

- **I3C** — наступник I²C (вища швидкість, зворотна сумісність на концептуальному рівні).
- **SMBus** / **PMBus** — похідні для живлення та моніторингу в ПК і серверах.
- **Clock stretching** — коли slave затримує SCL; практична причина таймаутів у драйверах.
- Типові адреси: EEPROM, RTC, сенсори температури/руху.

**Практичний контекст:** смартфони, плати розробки, монітори (DDC over I²C); **Pi:** `i2cdetect -y 1`; **ESP32/Arduino:** бібліотека `Wire`.

**Типові адреси:** 7-біт (наприклад OLED `0x3C`), pull-up 2.2–4.7 kΩ на SDA/SCL.

**Питання для самоперевірки (додаткові):**

1. Чому на шині I²C потрібні підтягуючі резистори?
2. Що таке clock stretching і коли воно виникає?

## Лекція 7. Інтерфейси Q-BUS, ISA, EISA

**Базовий матеріал:** магістралі міні- та персональних комп’ютерів (конспект 2025, лекція 7).

**Що додати / оновити:**

- Чітко позначити як **історичні** шини (DEC Q-BUS, ISA PC, EISA).
- Роль у курсі — **еволюція** від магістралі до PCIe; не сучасний стандарт для нових проєктів.
- ISA зникла з масових ПК наприкінці 1990-х — початку 2000-х.

**Практичний контекст:** музеї обчислювальної техніки, legacy-обладнання; у нових системах — PCIe.

**Питання для самоперевірки (додаткові):**

1. Яку проблему вирішила поява магістралі ISA порівняно з радіальними з’єднаннями?
2. Чому EISA не стала довготривалим стандартом масового ринку?

## Лекція 8. Інтерфейси PCI, PCI-E, AGP

**Базовий матеріал:** паралельна PCI, послідовна PCIe, слоти, конфігураційний простір (конспект 2025, лекція 8).

**Що додати / оновити:**

- **PCIe lanes** (x1, x4, x8, x16), **PCIe Gen4/Gen5/Gen6** (Gen6 — до ~64 ГТ/с на лінію, оглядово для серверів і нових платформ 2025–2026).
- **NVMe** — протокол накопичувачів поверх PCIe (заміна AHCI/SATA для SSD у ПК).
- **Мережеві карти (NIC)** — типовий периферійний пристрій на PCIe; драйвер NIC у ядрі, програми — через сокети (див. «Мережеві периферійні інтерфейси»).
- **Thunderbolt** — PCIe + DisplayPort по USB-C.
- **AGP** — застарілий; замінений PCIe для відеокарт.
- **CXL** (Compute Express Link) — пам’ять і accelerators (короткий огляд).

**Практичний контекст:** відеокарти, NVMe SSD, мережеві карти PCIe, FPGA-плати; **Raspberry Pi 5** — PCIe 2.0 x1 (FPC); **Jetson** — M.2 Key E/M для Wi‑Fi/NVMe модулів.

**Питання для самоперевірки (додаткові):**

1. Чому NVMe SSD швидший за SATA SSD?
2. Що означає позначення PCIe x4 Gen4?

## Лекція 9. BS-4421, Centronics, IEEE-488

**Базовий матеріал:** паралельні інтерфейси принтерів і лабораторних приладів (конспект 2025, лекція 9).

**Що додати / оновити:**

- **Centronics / паралельний порт принтера** — legacy; замінений USB і мережею.
- **IEEE-488 (GPIB)** — ще зустрічається в **лабораторному** вимірювальному обладнанні (осцилографи, джерела живлення).
- Для нових систем — USB, **Ethernet**, **LXI** (LAN для приладів; див. розділ «Мережеві периферійні інтерфейси»).

**Практичний контекст:** старі принтери; GPIB — університетські та промислові лабораторії зі старим парком приладів.

**Питання для самоперевірки (додаткові):**

1. Чому паралельні інтерфейси принтерів відійшли в минуле?
2. Де GPIB (IEEE-488) ще може бути актуальним у 2026 році?

## Лекція 10. ATA, SATA, SCSI

**Базовий матеріал:** IDE/ATA, SATA, SCSI, RAID (конспект 2025, лекція 10).

**Що додати / оновити:**

- **NVMe** vs **SATA** — основна відмінність для сучасних ПК.
- **SAS** — у серверах і СХД (замість паралельного SCSI).
- Стек драйверів: **AHCI** (SATA) vs **NVMe driver** (блочний рівень).
- **eMMC/UFS** — вбудована flash у мобільних (огляд).

**Практичний контекст:** SSD у ноутбуках (NVMe), HDD/SSD SATA у старіших системах, сервери SAS.

**Питання для самоперевірки (додаткові):**

1. Чому NVMe використовує PCIe, а не SATA?
2. У чому різниця між block device driver і файловою системою?

## Лекція 12. Драйвери зовнішніх запам’ятовуючих пристроїв

**Базовий матеріал:** flash, структура драйверів ЗЗП (конспект 2025, лекція 12).

## Лекція 14. Драйвери пристроїв виведення (текст і графіка)

**Базовий матеріал:** принтери, дисплеї, графічні адаптери (конспект 2025, лекція 14).

**Що додати / оновити:**

- **DisplayPort (DP)** і **HDMI** — основні **цифрові** інтерфейси виведення відео замість аналогового VGA (застарілий у нових системах).
- **HDMI 2.1** (огляд): до 48 Гбіт/с (FRL), 4K@120 Гц, 8K; часто в телевізорах і консолях; ліцензійна екосистема HDMI Forum.
- **DisplayPort 2.1** (огляд): до 80 Гбіт/с (UHBR); поширений у **моніторах і ноутбуках**; часто виводиться через **USB-C alt mode** (див. доповнення до лекції 2).
- **Порівняння (практично):**

| Ознака | HDMI | DisplayPort |
|--------|------|-------------|
| Типовий роз’єм на ПК/моніторі | HDMI-A | DP або USB-C (DP alt mode) |
| Аудіо разом із відео | так (CEC, eARC у TV) | так |
| Daisy-chain моніторів | ні (зазвичай) | так (Multi-Stream Transport, MST) |
| Зв’язок з USB-C | через адаптер / alt mode | нативний alt mode |

- Це **не** послідовний COM-порт і **не** шина типу USB для довільних пристроїв: відеоінтерфейс працює в парі з **GPU** (framebuffer, mode setting, EDID/DisplayID).
- Linux: **DRM/KMS** для графіки; **CUPS** для друку.
- **GDI / DirectX / Vulkan** — рівні графічного API (огляд, не деталі).

**Практичний контекст:** монітори через DP/HDMI або USB-C→DP; **мережеві принтери (IPP)** — див. також «Мережеві периферійні інтерфейси»; GPU-драйвери від NVIDIA/AMD/Intel. Окремої лабораторної на DP/HDMI чи Ethernet у практикумі немає.

**Питання для самоперевірки (додаткові):**

1. Чим драйвер дисплея відрізняється від відеодрайвера GPU?
2. Яку роль відіграє CUPS у Linux при друці?
3. Чому HDMI і DisplayPort не включені в лабораторний практикум разом із RS-232 та USB?
4. Як ноутбук може виводити зображення на зовнішній монітор через один порт USB-C?
